# 02 启动与构建链路注意事项

## 1. 启动链路（`markcompose.sh start`）

根目录只保留一个入口 `markcompose.sh`，启动子命令的真实实现拆到了 `scripts/start.sh` 与 `scripts/lib/`。

执行顺序：

1. 参数校验（路径、端口、命令依赖）
2. 准备 editor（默认包或自定义目录）
3. 准备 watcher（默认包或自定义命令）
4. 写 `.env.runtime`
5. 执行一次 `markcompose.sh build`
6. 启动 `nginx`，并按需启动 `markwatch`

## 2. 强约束（违反会直接失败）

- 必须有且仅有一个 `<markdown_dir>` 位置参数
- `HOST_PORT` 与 `EDITOR_PORT` 必须不同
- 路径不允许包含空白字符（`MARKDOWN_DIR`、`EDITOR_STATIC_DIR`、`ADAPTER_SCRIPT`）
- `assets-dir` 必须是单级目录名（不能带 `/`，不能是 `.`/`..`）
- 默认 watcher 仅支持 Linux，CPU 仅 `amd64`/`arm64`

## 3. 构建链路（`markcompose.sh build`）

根目录只保留一个入口 `markcompose.sh`，构建子命令的真实实现拆到了 `scripts/build.sh`。

构建链路仍然是“临时目录构建 + gate + 发布替换”：

1. （可选）先运行 content adapter，输出到 `.runtime/content-adapted`
2. 构建到临时 staging 目录
3. gate：必须存在 `index.html`
4. 清空 `hugo_public` 并复制 staging 产物

注意：

- `markcompose.sh build` 依赖 `.env.runtime`，通常需先执行 `markcompose.sh start`
- 如果启用 adapter，`ADAPTER_SCRIPT` 必须存在且可执行

## 4. 停止链路（`markcompose.sh stop`）

根目录只保留一个入口 `markcompose.sh`，停止子命令的真实实现拆到了 `scripts/stop.sh`。

- 默认：`docker compose down`
- `-v`：`docker compose down --volumes`
- 额外会尝试停止 `.markwatch.pid` 中记录的 watcher 进程

## 5. 常见运行风险

- watcher 启动失败只会告警，主服务仍可能启动成功；要看 `.markwatch.log`
- 自定义 watcher 命令建议保持单前台命令，避免 `markcompose.sh stop` 难以清理子进程
- 首次拉默认资源依赖网络；下载失败会中断启动
