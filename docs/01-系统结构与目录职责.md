# 01 系统结构与目录职责

## 1. 运行形态（双端口静态服务）

- 博客站点：`http://127.0.0.1:<HOST_PORT>/`（默认 `8080`）
- 编辑器站点：`http://127.0.0.1:<EDITOR_PORT>/`（默认 `8081`）
- 资源目录：`http://127.0.0.1:<HOST_PORT>/<ASSETS_DIR>/...`（默认 `/_assets/`）

核心是“同一个 `nginx` 容器，监听 80/81 两个端口，分别挂载博客输出与编辑器静态目录”。

## 2. Docker Compose 角色分工

- `hugo-builder`：
  - 使用 `docker.io/hugomods/hugo:dart-sass-non-root`
  - 读取 `MARKDOWN_DIR`（只读）
  - 读取 `./hugo-site`（主题与布局）
  - 写入 `hugo_public` volume（博客发布产物）
- `nginx`：
  - 将 `hugo_public` 挂载为 `/var/www/blog`
  - 将 `EDITOR_STATIC_DIR` 挂载为 `/var/www/editor`
  - 配置文件来自 `./nginx/default.conf`

## 3. 关键目录职责

- `hugo-site/`：Hugo 站点、主题、菜单配置、布局模板
- `hugo-reuse/layouts/`：复用的 Hugo layouts / partials / render hooks，会在构建前复制到 `hugo-site/layouts/`
- `adapter/`：可选内容适配器脚本、Python 实现与配置
- `scripts/`：`markcompose.sh` 子命令的真实实现，以及 `scripts/lib/` 公共 shell 模块
- `.runtime/`：默认 editor/watcher 下载包及解压内容、适配后内容输出
- `markdown_test/`：示例 markdown 工作区

## 4. 关键运行文件

- `markcompose.sh`：根目录唯一入口，统一承载 `start/build/stop` 子命令
- `.env.runtime`：`markcompose.sh start` 生成的运行时环境参数源
- `.markwatch.pid` / `.markwatch.log`：watch 进程 PID 和日志

## 5. 容易忽略的架构点

- 这个项目不是 `hugo server` 模式，而是“构建后静态托管”模式。
- `markcompose.sh build` 是发布流水线（带 gate + replace publish）；具体实现拆在 `scripts/`。
- `hugo_public` 是命名卷，不在仓库目录里；排查“页面没更新”要考虑卷里内容是否替换成功。
